# Toy Dataset Cards

These cards describe the generated toy datasets used by the SARN-Dense Phase 1/2 baseline. They are engineering fixtures for correctness, reproducibility, and variance checks. They are not real language benchmarks and should be replaced before any claim about natural-language capability.

SARN-Dense is the baseline/control for these fixtures, not a useful natural-language model.

## `toy/repeated_pattern`

- Dataset name: `toy/repeated_pattern`
- Purpose: verify that training, evaluation, checkpointing, generation, and reporting work on a tiny deterministic next-token task.
- Generation method: repeat a short integer pattern, currently `(1, 2, 3, 4)`, then use each token as input and the next token as the label.
- Train/eval split: smoke training and toy validation both generate fresh batches from the same deterministic rule; they are not independent corpus splits.
- Expected behavior: a small SARN-Dense model should reduce loss quickly and achieve high token accuracy after smoke overfitting.
- Limitations: the task has very low entropy, no semantics, no long-range language structure, and no contamination risk representative of real data.
- Why this is not a real language benchmark: the model can solve it by memorizing a short periodic transition table.
- Future replacement: a governed tokenizer and real held-out text or structured benchmark suite with documented licenses, splits, deduplication, and contamination checks.

## `toy/copy`

- Dataset name: `toy/copy`
- Purpose: provide a simple synthetic sequence-copy fixture for future controlled generalization tests.
- Generation method: sample token payloads from a seeded generator, insert delimiter-like high token IDs, and repeat the payload.
- Train/eval split: generated by seed; different seeds can create different toy variants, but this is still synthetic data.
- Expected behavior: models with enough context and training should learn short exact copying under the generated format.
- Limitations: no natural language, no semantic ambiguity, and no realistic distribution shift.
- Why this is not a real language benchmark: success measures format copying, not comprehension or useful generation.
- Future replacement: structural generalization tasks with frozen generators, train/validation/test seeds, and matched dense baselines.

## `toy/byte_text`

- Dataset name: `toy/byte_text`
- Purpose: exercise the byte tokenizer and generation path with printable bytes.
- Generation method: repeat a fixed short text string and slice shifted windows into inputs and labels.
- Train/eval split: deterministic windows over a repeated sentence; it is not a corpus split.
- Expected behavior: the model may emit recognizable byte fragments after overfitting the tiny string.
- Limitations: the repeated text is too small to evaluate language modeling quality or factual behavior.
- Why this is not a real language benchmark: generated output reflects a tiny repeated byte stream, not learned natural-language ability.
- Future replacement: licensed text datasets with stable preprocessing, train/eval/test splits, data cards, and baseline reports.

## Governance Notes

- All toy datasets are generated locally and require no network access.
- Metrics from these datasets are suitable for regression tests and reproducibility checks only.
- Reports must state that toy-byte generation is not useful natural-language output.
- Future datasets should include source, license, preprocessing, split policy, deduplication, contamination checks, and intended replacement criteria.
